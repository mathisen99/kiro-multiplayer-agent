# Requirements Document

## Introduction

Launch Room is a one-day MVP for collaborative product planning. Two people use a
shared, anonymous workspace to turn a rough idea into a concise Markdown plan. An
invited Product Agent may propose cards, but people must approve AI-authored
content before it can appear in the final artifact.

The required demo journey is the product boundary. Reliability and clarity in
that journey take priority over additional features, architectural abstraction,
visual polish, and broad automated test coverage.

## Glossary

- **Workspace**: A room reached through an unguessable shared URL.
- **Participant**: A person who creates or joins a Workspace with a nickname and
  no account.
- **Client ID**: A UUID generated by the browser and stored in local storage. It
  identifies the browser within this demo; it is not authentication.
- **Card**: A planning item in one of four sections: `problem`, `requirements`,
  `risks`, or `tasks`.
- **Human Card**: A Card created by a Participant. Its status is `active`.
- **Product Agent**: The single AI role available in the MVP.
- **AI Proposal**: A Card created by the Product Agent. Its status is initially
  `proposed` and it may include a rationale and source Card references.
- **Eligible Card**: An `active` Human Card or an `approved` AI Proposal.
- **Final Artifact**: The stored Markdown plan generated only from Eligible
  Cards.
- **Safe Error**: A concise error message that contains no secret, full room
  snapshot, stack trace, or verbose AI-provider payload.

## Requirements

### Requirement 1: MVP Boundary

**User Story:** As the builder, I want a strict scope boundary so that I can
complete a reliable one-day challenge submission.

#### Acceptance Criteria

1. THE Launch Room system SHALL implement the Required Demo Journey before any
   optional feature.
2. WHEN the Required Demo Journey succeeds and the Validation Gate passes, THE
   Launch Room system SHALL consider the MVP complete.
3. WHILE either condition in Requirement 1.2 has a blocker, THE implementation
   SHALL limit work to resolving that blocker.
4. THE Launch Room system SHALL omit every feature listed under Explicit
   Exclusions.

### Requirement 2: Workspace Creation and Joining

**User Story:** As a Participant, I want to create and share a room without an
account so that another person can join immediately.

#### Acceptance Criteria

1. WHEN a person submits a room name of 1-100 characters, an optional goal of at
   most 500 characters, a rough idea of 1-900 characters, and a nickname of
   1-50 characters, THE system SHALL create a Workspace and redirect to
   `/room/[roomId]`.
2. WHEN a Workspace is created, THE system SHALL create the rough idea as its
   first Human Card in the `problem` section.
3. WHEN a new browser opens an existing Workspace URL, THE system SHALL ask for
   a nickname of 1-50 characters and join without account credentials.
4. WHEN a browser creates or joins a Workspace, THE client SHALL store its Client
   ID and Workspace nickname in local storage.
5. THE room page SHALL provide a control that copies the current Workspace URL.
6. THE home page SHALL state `No account required for this demo`.
7. IF any creation or join input is invalid, THEN THE system SHALL show a Safe
   Error and SHALL NOT create or modify Workspace records.
8. IF a requested Workspace does not exist, THEN THE system SHALL show a Safe
   Error and a route back to the home page.

### Requirement 3: Shared Planning Board

**User Story:** As a Participant, I want to add and edit planning cards so that
the board reflects the team's decisions.

#### Acceptance Criteria

1. THE room page SHALL display exactly four Card sections named `Problem`,
   `Requirements`, `Risks`, and `Tasks`.
2. WHEN a Participant submits a section, a title of 1-100 characters, and content
   of 1-900 characters, THE system SHALL create a Human Card with status
   `active` and server-assigned authorship.
3. WHEN a Participant edits a Human Card in the same Workspace using valid input,
   THE system SHALL update its section, title, content, and update timestamp.
4. THE system SHALL NOT offer editing controls for AI Proposals.
5. IF a Card input is invalid, the Card does not exist, or the Card belongs to a
   different Workspace, THEN THE system SHALL reject the mutation with a Safe
   Error and SHALL NOT modify any Card.
6. THE server SHALL ignore client-supplied IDs, authorship, status, ownership, and
   timestamps except for the Client ID used to find the current Participant.

### Requirement 4: Two-Browser Synchronization

**User Story:** As a Participant, I want another participant's changes to appear
automatically so that the room feels shared during the demo.

#### Acceptance Criteria

1. WHEN a joined Participant opens the room page, THE client SHALL fetch the
   current room snapshot immediately.
2. WHILE the room tab is visible, THE client SHALL poll for the current snapshot
   approximately every 1-2 seconds without overlapping requests.
3. WHEN a local mutation succeeds or a hidden room tab becomes visible, THE
   client SHALL fetch a fresh snapshot immediately.
4. WHEN a valid snapshot arrives, THE client SHALL replace its local room state
   and order Cards by section, creation time, and Card ID.
5. IF polling fails or returns an invalid snapshot, THEN THE client SHALL retain
   the last valid state, display `Reconnecting`, and retry with delays of about
   1.25, 2.5, and at most 5 seconds.
6. WHEN polling succeeds, THE client SHALL reset the normal polling interval and
   display `Live`.
7. WHEN one of two visible browser sessions successfully changes a Card, THE
   other session SHALL show that change without manual refresh.

### Requirement 5: Product Agent

**User Story:** As a Participant, I want to deliberately run one AI teammate so
that it can suggest improvements without taking control of the plan.

#### Acceptance Criteria

1. THE system SHALL expose the Product Agent as the only runtime AI role.
2. WHEN a Participant invites the Product Agent, THE system SHALL add it to the
   Workspace without making an AI-provider request.
3. WHEN a Participant selects `Run teammate` and supplies an instruction of
   1-300 characters, THE system SHALL create one Agent Run and make one bounded
   AI-provider request.
4. THE Product Agent SHALL use only the supplied instruction and a fresh snapshot
   containing the room name, goal, active Human Cards, approved AI Proposals, and
   the Card IDs needed for source references.
5. THE Product Agent SHALL focus on clarifying the problem, proposing small and
   testable requirements or tasks, identifying important risks or missing
   decisions, and protecting the one-day scope.
6. WHEN an Agent Run succeeds, THE system SHALL create two or three AI Proposals
   on the board rather than chat messages.
7. EACH AI Proposal SHALL contain a valid section, a title of 1-100 characters,
   content of 1-900 characters, a rationale of 1-300 characters, and no more
   than four source Card IDs.
8. BEFORE storing AI output, THE server SHALL validate its structure, discard
   source IDs that are not in the current Workspace, and assign ownership,
   authorship, IDs, timestamps, and `proposed` status from server state.
9. THE system SHALL insert all proposals and mark the Agent Run complete in one
   database transaction.
10. IF the instruction is invalid, the AI request fails, the output is invalid,
    or the transaction fails, THEN THE system SHALL preserve existing Cards,
    mark the run failed when possible, display
    `Teammate could not complete this run`, and allow retry.

### Requirement 6: Human Review of AI Proposals

**User Story:** As a Participant, I want to review every AI contribution so that
people retain final authority.

#### Acceptance Criteria

1. THE system SHALL display each proposed AI Card with the Product Agent name and
   role, a `Proposed` badge, title, content, rationale, valid source links, and
   `Approve` and `Reject` controls.
2. WHEN a Participant approves a proposed AI Card in the current Workspace, THE
   system SHALL change its status from `proposed` to `approved`.
3. WHEN a Participant rejects a proposed AI Card in the current Workspace, THE
   system SHALL change its status from `proposed` to `rejected` and hide it from
   the default board.
4. IF the Card is not a proposed AI Card in the current Workspace, THEN THE
   system SHALL reject the review action with a Safe Error and preserve its
   status.
5. THE system SHALL preserve all Human Cards while applying or reviewing AI
   output.

### Requirement 7: Final Artifact and Export

**User Story:** As a Participant, I want a final Markdown plan based only on
accepted content so that pending or rejected suggestions cannot enter the
deliverable.

#### Acceptance Criteria

1. WHEN a Participant requests finalization, THE server SHALL send only Eligible
   Cards to the AI provider and SHALL exclude proposal rationales.
2. WHEN finalization succeeds, THE system SHALL store a title of 1-120 characters
   and Markdown of 100-10,000 characters.
3. THE generated Markdown SHALL contain `# Title`, `## Summary`, `## Problem`,
   `## Target user`, `## Goals`, `## Non-goals`, `## Requirements`,
   `## Acceptance criteria`, `## Risks and open questions`,
   `## Implementation plan`, and `## Demo checklist`.
4. THE Final Artifact SHALL preserve unresolved questions as unresolved, omit
   unsupported facts and internal AI details, and remain suitable for a one-day
   MVP.
5. WHEN a Final Artifact exists, THE system SHALL render the stored Markdown with
   `react-markdown` and without `dangerouslySetInnerHTML`.
6. THE copy control SHALL copy the complete stored Markdown without modification.
7. THE download control SHALL download the complete stored Markdown as
   `launch-room-<title-slug>.md`.
8. IF generation, validation, storage, or preview fails, THEN THE system SHALL
   show a Safe Error, retain the previous stored artifact and all Cards, and
   provide a retry path. A preview failure SHALL NOT disable copy or download.

### Requirement 8: Data, API, and Trust Boundaries

**User Story:** As the builder, I want small validated server boundaries and
simple persistence so that malformed or cross-room requests cannot corrupt the
demo.

#### Acceptance Criteria

1. THE system SHALL run as one local Next.js Node process using SQLite through
   `better-sqlite3` and prepared SQL statements, without an ORM.
2. THE system SHALL use the four tables `rooms`, `participants`, `cards`, and
   `agent_runs`; the stored Final Artifact SHALL be held on the relevant room
   record.
3. THE database helper SHALL use one process-wide connection, apply an idempotent
   schema, enable WAL and foreign keys, and set a 5-second busy timeout.
4. THE database path SHALL come from `DATABASE_PATH` and default to
   `./data/launch-room.db`; the system SHALL create its parent directory when
   necessary and ignore SQLite runtime files in version control.
5. THE system SHALL provide route handlers for room creation and retrieval,
   joining, Card creation and editing, Product Agent invitation and execution,
   proposal review, and finalization. It SHALL NOT provide a delete route.
6. EVERY mutation route SHALL validate input before writing, verify Workspace and
   referenced-record ownership, and apply its writes in one transaction.
7. THE server SHALL generate persisted IDs and timestamps and derive ownership,
   authorship, roles, and statuses from server state.
8. IF validation, ownership checks, or a transaction fails, THEN THE route SHALL
   return a Safe Error and SHALL NOT partially apply the mutation.
9. THE server SHALL keep `OPENAI_API_KEY` out of client code, responses, and
   logs, and SHALL NOT log complete room snapshots or verbose AI payloads.
10. THE README SHALL explain that anonymous links and Client IDs are demo
    conveniences, not production authorization, and that the application is not
    suitable for confidential data, serverless deployment, or multiple app
    instances.

### Requirement 9: Technology and AI Baseline

**User Story:** As the builder, I want a small fixed technology baseline so that
implementation time is not lost to unnecessary choices.

#### Acceptance Criteria

1. THE system SHALL use Next.js App Router, React, TypeScript, Tailwind CSS, Zod,
   `better-sqlite3`, the official OpenAI JavaScript SDK, `react-markdown`, and
   pnpm with a committed `pnpm-lock.yaml`.
2. THE implementation SHALL pin compatible dependency versions in
   `package.json`; detailed version selection belongs in the design rather than
   in product behavior requirements.
3. THE server SHALL read the AI model from `OPENAI_AGENT_MODEL` and SHALL fail
   with a Safe Error when the key or model configuration is unavailable.
4. THE system SHALL use the OpenAI Responses API with structured output and
   `store: false` for Agent Runs and finalization.
5. THE system SHALL use low reasoning effort for Agent Runs and medium reasoning
   effort for finalization, when supported by the configured model.
6. THE system SHALL use OpenAI as its only external runtime service.
7. THE runtime SHALL NOT use hosted tools, the Assistants API, the Agents SDK,
   MCP, multi-agent orchestration, background mode, autonomous loops, Kiro CLI,
   or arbitrary command execution.

### Requirement 10: Demo-Ready Experience and Validation

**User Story:** As a demo viewer, I want a clear and reliable planning workspace
so that the human-AI collaboration is understandable in a short recording.

#### Acceptance Criteria

1. THE home page SHALL show `Launch Room`, a one-line explanation, the creation
   fields from Requirement 2, and a `Create workspace` control.
2. THE room page SHALL show the room name, goal, connection status, share control,
   participants, Product Agent controls, four-section board, Card actions, and
   Final Artifact controls.
3. THE board SHALL be the primary desktop visual area and use a restrained,
   high-contrast interface with clear empty, loading, proposal, and error states.
4. DURING a recoverable synchronization, AI, database, or finalization failure,
   THE system SHALL keep the last valid Workspace content visible and offer a
   retry path.
5. THE project SHALL define `check:quick` as `tsc --noEmit`, `check` as
   `pnpm run check:quick && pnpm run lint`, lint as `eslint .`, and `verify` as
   `pnpm run check && pnpm run build`.
6. THE Validation Gate SHALL consist of `pnpm run verify` and one successful
   manual run of the Required Demo Journey in two browser sessions.
7. THE MVP SHALL NOT require a browser automation suite, broad unit-test suite,
   coverage tooling, load testing, or model evaluations.
8. IF a small pure function contains non-trivial critical logic, THEN the project
   MAY include at most two focused unit tests without adding a test framework
   solely for them.

## Required Demo Journey

The manual acceptance scenario SHALL be completed in this order:

1. Create a Workspace named `Weekend Launch` with goal
   `Plan a one-day MVP for a local event discovery app`, rough idea
   `Users choose a city and interests, then receive a short weekend plan`, and
   nickname `Alex`.
2. Confirm the redirect and initial Problem Human Card.
3. Open the Workspace URL in a private or incognito browser session and join as
   `Sam`.
4. Add a Requirement Human Card as Sam.
5. Confirm Alex sees Sam's Card without manual refresh.
6. Invite the Product Agent.
7. Run it with `Turn this into the smallest testable user flow.`
8. Confirm two or three AI Proposals appear on the board.
9. Approve one AI Proposal and reject another.
10. Generate the Final Artifact.
11. Confirm active Human Cards and the approved AI Proposal are included.
12. Confirm proposed and rejected AI Proposals and proposal rationales are
    excluded.
13. Preview, copy, and download the Markdown.
14. Run `pnpm run verify`.

## Kiro Project Artifacts

The implementation work SHALL also produce:

- `.kiro/specs/launch-room/design.md` and `tasks.md`.
- Concise always-included Steering files at `.kiro/steering/product.md`,
  `tech.md`, and `structure.md`.
- File-matched agent guidance for agent and finalization server code.
- One post-task quality hook that runs `pnpm run check:quick`.
- One demo-readiness skill that checks only submission blockers, including the
  Required Demo Journey, artifact eligibility, export behavior, and the
  Validation Gate.
- A short, honest Kiro demo checklist. It SHALL NOT claim that a person performed
  a Kiro UI action unless that action actually occurred.

`tasks.md` SHOULD contain 8-10 dependency-ordered tasks and track completion as
implementation proceeds. These artifacts should repeat only the constraints
needed in their own context, not this entire document.

## Explicit Exclusions

- Accounts, private rooms, or production authorization
- Persistent chat, comments, or agent-to-agent conversation
- Additional AI roles, autonomous runs, or scheduled runs
- Drag-and-drop, rich-text collaboration, or offline conflict resolution
- File uploads, GitHub integration, runtime MCP, or arbitrary command execution
- Notifications, analytics, billing, or voice
- Version history, deletion, or editing approved AI Proposals
- Production rate limiting or confidential-data support
- WebSockets, Server-Sent Events, Socket.IO, Redis, hosted synchronization, or
  hosted database services
- ORMs, multi-instance replication, serverless deployment, or repository/service
  architecture layers
- Broad automated test suites and optional enhancements outside this document

## Definition of Done

The MVP is done when the Required Demo Journey works reliably, `pnpm run verify`
passes, the required Kiro project artifacts exist, the UI is suitable for a short
recorded demo, and no excluded feature has been added.
